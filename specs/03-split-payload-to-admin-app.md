# Spec 03 — Split Payload CMS into `apps/admin/`

## Context

Currently `apps/rfe-v0/` is a single Next.js app with Payload CMS integrated. The agency workflow requires Payload to live in a **separate app** (`apps/admin/`), so that:

1. The v0 site stays lightweight (no CMS overhead)
2. Payload is only added when the client pays for CMS
3. Content is progressively migrated via the admin's REST API

## Current state

### Payload wiring in `apps/rfe-v0/`

| File | Purpose |
|------|---------|
| `next.config.mjs` | `withPayload()` wrapper |
| `payload.config.ts` | Calls `buildRfeConfig()` from `@rfe/cms/config` |
| `app/(payload)/layout.tsx` | Admin shell (RootLayout, ServerFunctions) |
| `app/(payload)/custom.css` | Admin Tailwind preflight skip |
| `app/(payload)/admin/[[...segments]]/page.tsx` | Admin UI catch-all |
| `app/(payload)/admin/importMap.js` | Auto-generated import map |
| `app/(payload)/api/[...slug]/route.ts` | REST API catch-all |
| `scripts/seed/index.ts` | Seed entry calling `@rfe/cms/seed` |

### Payload deps in `apps/rfe-v0/package.json`

`payload`, `@payloadcms/graphql`, `@payloadcms/next`, `@payloadcms/storage-s3`, `@payloadcms/ui`, `@rfe/cms`

### Frontend Payload usage: **NONE**

No `payload.find()`, `payload.findGlobal()`, or `getPayload()` calls in `app/(frontend)/`. The frontend is 100% hardcoded. This makes the split clean — no data-fetching rewiring needed.

- **Port 3001** for dev (v0 frontend stays on 3000)

- **Seed** delegates to `@rfe/cms/seed` (same pattern as current v0)

## Plan D — Typed API client in `@rfe/cms` (optional, independent)

**Can run anytime.** Only useful once content migration begins (v0 needs to fetch from admin REST API).

### What to add

**`packages/cms/src/client.ts`** — typed REST client factory:
```typescript
import type { Config } from './payload-types'

export function createPayloadClient(baseUrl: string) {
  return {
    find: async <T extends keyof Config['collections']>(
      collection: T,
      query?: { locale?: string; limit?: number; where?: Record<string, unknown> }
    ) => {
      const params = new URLSearchParams()
      if (query?.locale) params.set('locale', query.locale)
      if (query?.limit) params.set('limit', String(query.limit))
      const res = await fetch(`${baseUrl}/api/${collection}?${params}`)
      return res.json()
    },
    findGlobal: async (slug: string, query?: { locale?: string }) => {
      const params = new URLSearchParams()
      if (query?.locale) params.set('locale', query.locale)
      const res = await fetch(`${baseUrl}/api/globals/${slug}?${params}`)
      return res.json()
    },
  }
}
```

**Generate `payload-types.ts`:**
```bash
pnpm --filter @rfe/admin generate:types
# Copy or symlink to packages/cms/src/payload-types.ts
```

**Export from `packages/cms/package.json`:**
```jsonc
"./client": "./src/client.ts"
```

### Usage in v0 (when content migration starts)

```typescript
import { createPayloadClient } from '@rfe/cms/client'

const cms = createPayloadClient(process.env.CMS_API_URL || 'http://localhost:3001')
const works = await cms.find('works', { locale: 'en' })
```

### Tasks

1. Generate `payload-types.ts` from admin app
2. Create `packages/cms/src/client.ts`
3. Add `./client` export to `packages/cms/package.json`
4. Verify types compile

**Plans A and B touch completely different files** (A creates new files in `apps/admin/`, B modifies existing files in `apps/rfe-v0/`). They can safely run in parallel.

**Plan D** is independent and can run alongside A+B, but is only useful once content collections exist beyond Users/Media.

**Plan C** must wait for A+B because it runs `pnpm install` and `pnpm build` across the whole monorepo.
